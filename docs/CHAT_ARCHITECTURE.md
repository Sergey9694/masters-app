# 💬 Архитектура Чата и Уведомлений (Фаза 7)

Документ описывает техническую реализацию системы обмена сообщениями и real-time уведомлений в проекте **УслугиРядом**.

---

## 1. Технологический стек
- **Socket.io 4.7+**: Двусторонний обмен данными.
- **Redis (ioredis)**: Адаптер для масштабирования WebSocket-соединений между несколькими инстансами сервера.
- **Next.js (Custom Server)**: Использование `server.ts` для инициализации сокетов.
- **Prisma + PostgreSQL**: Постоянное хранение сообщений.
- **AES-256-GCM**: Сквозное шифрование текста сообщений в базе данных.

## 2. Модель данных
- **Conversation**: Комната чата. Связывает участников. Может быть привязана к конкретному заказу (`orderId`).
- **ConversationParticipant**: Связующая таблица для участников и отслеживания последнего прочтения (`lastReadAt`).
- **Message**: Сообщение. Хранит зашифрованный текст, ссылки на вложения и метаданные удаления.

## 3. Безопасность и Валидация
### 3.1. Защита WebSocket (Middleware)
- Проверка сессии `Auth.js` при каждом `connect`.
- **UUID Валидация**: Сокеты принимают только валидные UUID для ID комнат.
- **Participant Validation**: Пользователь не может подписаться на события (join) или отправить сообщение в комнату, участником которой он не является.

### 3.2. Rate Limiting
- Ограничение: **30 сообщений в 60 секунд**.
- Реализовано на уровне API и Socket-эмиттеров. Предотвращает спам и DoS-атаки.

### 3.3. CORS и CSP
- CORS ограничен только доверенным доменом из `NEXTAUTH_URL`.
- CSP политика разрешает соединения `ws:` и `wss:`.

## 4. Real-time уведомления
Система автоматически отправляет события через сокеты при:
- Новом сообщении (`NEW_MESSAGE`).
- Изменении статуса прочтения.
- Системных уведомлениях (новый заказ, новый отклик).

Если пользователь оффлайн, уведомление сохраняется в БД и будет показано при следующем входе.

## 5. Производительность
- **Виртуализация**: Список сообщений использует `react-virtuoso` для плавной работы при тысячах сообщений.
- **Пагинация**: Сообщения загружаются порциями по 50-200 штук.
- **Оптимизация счетчиков**: Счетчик непрочитанных сообщений вычисляется через эффективный `count` запрос в БД, а не через загрузку всех объектов.

## 7. Система присутствия (Online Presence)
- **Хранение**: `lastSeenAt` в БД + временный ключ в Redis (`user:status:{userId}`) с TTL.
- **Синхронизация**: При `connection` сокета устанавливается "online", при `disconnect` (через 3с задержки) — "offline".
- **Real-time**: Событие `user:status` рассылается всем участникам общих бесед.
- **UI**: 
  - Зеленая точка в `ConversationHeader` (рядом с именем) и в `ConversationList` (рядом с аватаром).
  - Индикатор "печатает..." принудительно сбрасывается при получении сообщения от собеседника.
  - Статусы в списке чатов инициализируются серверно (`isOnline`) и обновляются через сокеты.

## 6. Развертывание (Production)
- Сервер запускается через `tsx server.ts` (скрипт `startup.js`).
- Требуется запущенный инстанс **Redis** (настроен в `docker-compose.yml`).
- Переменная `NEXTAUTH_URL` обязательна для корректной работы CORS.
