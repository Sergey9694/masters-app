## Как давать задачи

### Простая задача
Просто пишешь что нужно:
```
"добавь валидацию в форму создания объявления"
"почему 404 на странице my-orders?"
"как лучше реализовать пагинацию в каталоге?"
```

### Сложная / многошаговая задача
Добавляй "как оркестратор":
```
"как оркестратор — доделай фазу 6"
"как оркестратор — проведи полный аудит безопасности"
"/delegate подготовь проект к Фазе 7 (чат)"
```
Оркестратор работает по `.agent/workflows/delegate.md`: разобьёт задачу на подзадачи, покажет план, запросит одобрение, выполнит роли через нужные скиллы и пройдет финальные проверки.

### Вызов конкретного агента
```
"как архитектор — проверь структуру features/listing-management"
"как security-агент — проверь auth-логику"
"как мобильный эксперт — проверь UX на 375px"
"как мастер монорепо — вынеси логику в shared"
"как devops — проверь логи в Sentry"
"как QA — какие edge-cases я не учёл в форме заказа?"
```

## Файлы которые нужно знать

| Файл | Зачем | Когда читать |
|------|-------|-------------|
| `AGENTS.md` | Входной файл для Codex | Начало Codex-сессии |
| `docs/PROJECT_STATE.md` | Текущее состояние проекта | Начало каждой сессии |
| `DEVELOPMENT_PLAN.md` | Полный план всех фаз | Когда нужны детали задачи |
| `.agent/workflows/delegate.md` | Канонический workflow `/delegate` | Для сложных/многошаговых задач |
| `.agent/skills/agent-protocol/SKILL.md` | Правила для всех агентов | При настройке / отладке |
| `docs/agent-reports/` | Отчёты агентов | После аудитов |

## Как обновлять PROJECT_STATE.md

После каждой значимой задачи агент должен обновить `docs/PROJECT_STATE.md`.
Если не обновил — попроси явно:
```
"обнови PROJECT_STATE.md"
```

Что обновлять вручную (если агент забыл):
- `🕓 Последнее обновление` — дата
- Статус фазы в таблице (🟡 → ✅)
- Раздел "Последние значимые изменения" — добавь 1 строку
- Раздел "Известные TODO" — убери закрытые

## Правила которые агенты НИКОГДА не нарушают

1. **📚 Docs First** — Сначала проверяют актуальную документацию библиотек, потом пишут код.
2. **🔒 Security Gate** — runtime-код сначала проверяет безопасник; CRITICAL/HIGH риск блокирует работу.
3. **💰 Экономия токенов** — не читают файл целиком если нужна одна строка.
4. **🗄️ БД** — любое изменение schema.prisma → `prisma migrate dev` → сначала спросить тебя.
5. **🔐 Frontend is untrusted** — секреты, доступы и проверки безопасности только на backend.
6. **🏆 Рекомендация** — всегда дают рекомендацию "Вариант N лучше, потому что..."
7. **🏗️ FSD & Clean Architecture** — соблюдают иерархию слоев и чистоту зависимостей.
8. **🧪 Проверка** — после каждого изменения `tsc --noEmit` и тесты.

Скажи:

```
"прочитай раздел 'Фаза 6' в DEVELOPMENT_PLAN.md"
```
