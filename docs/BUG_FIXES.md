## [2026-04-22] Исправление регистрации мастера и ленты заказов

### 1. Дублирование окна выбора файлов (Double File Dialog)
**Описание:** При нажатии на зону загрузки портфолио в форме регистрации мастера окно выбора файлов открывалось дважды.
**Причина:** 
- Компонент `Field` был реализован через тег `<label>`, внутри которого находился `div` с обработчиком `onClick`, вызывающим `input.click()`. 
- В HTML клик по `label` автоматически триггерит связанный `input`. В итоге срабатывал и JS-обработчик, и нативное поведение браузера.
**Решение:**
- Корневой элемент `Field` изменен с `label` на `div` (файл `ProviderRegistrationFormLight.tsx`).
- Добавлено состояние `isProcessingFiles` для блокировки повторных кликов во время обработки (HEIC -> JPEG).

### 2. Пустая лента заказов для новых мастеров (Smart Feed Lock)
**Описание:** Мастера после регистрации видели пустую ленту, если в их категориях еще не было заказов. Кнопка «Все категории» не помогала, так как сбрасывала фильтр в URL, что снова включало «Умную ленту» по категориям профиля.
**Причина:** 
- В `order.service.ts` логика была жестко завязана: если `categoryId` отсутствует — фильтруем по профилю мастера. 
- В интерфейсе отсутствовал способ явно запросить «всё без исключений».
**Решение:**
- Введен зарезервированный параметр `all` для `categoryId`.
- Обновлен `OrdersFilters`: добавлен пункт «✨ Моя лента» (сброс в умный дефолт) и «Все категории» (параметр `all`).
- В `OrderFeedClient` добавлены «умные заглушки»: если заказов нет, пользователю предлагается кнопка «Посмотреть все активные заказы», которая принудительно отключает фильтры.

### 3. Медленная обратная связь при загрузке фото
**Описание:** При выборе тяжелых фото (iPhone HEIC) казалось, что ничего не происходит, пока шла конвертация на клиенте.
**Решение:** 
- Внедрен индикатор загрузки с текстом «Обработка изображений...» в реальном времени.

## [2026-04-22] Внедрение SEO-ссылок и порядковых номеров заказов

### 1. Переход на SEO-дружелюбные URL (Slugs)
**Описание:** Ранее ссылки на заказы содержали технические ID (CUID), например `/orders/cmoa...`. Это плохо для SEO и неудобно для пользователей.
**Решение:**
- В модель `Order` добавлены поля `orderNumber` (автоинкремент) и `slug` (уникальная строка).
- Создана утилита `slugify.ts` для транслитерации кириллицы.
- При создании заказа генерируется слаг формата `nazvanie-zakaza-123`.
- Роутинг изменен с `orders/[id]` на `orders/[slug]`.
- Реализована обратная совместимость: поиск заказа в БД теперь идет по ID или по слагу (через `OR`).

### 2. Порядковые номера заказов (Human-friendly IDs)
**Описание:** У заказов не было короткого идентификатора для общения с поддержкой или вывода в статистику.
**Решение:**
- Использован тип `Int @unique @default(autoincrement())` в Postgres.
- Номера заказов выведены во всех ключевых местах: в ленте, в списке «Мои заказы» и на детальной странице заказа в формате `№123`.

## [2026-04-22] Иерархическая SEO-маршрутизация и структура категорий

### 1. Переход на структуру /orders/[categorySlug]/[slug]
**Описание:** Одиночные слаги `/orders/[slug]` не давали поисковым системам понимания тематической вложенности заказа.
**Решение:**
- [x] **[P2022] Column `Order.executionDate` does not exist**: Ошибка возникала из-за наличия поля в схеме Prisma при его отсутствии в базе данных на сервере. Поле `executionDate` временно удалено из всех слоев приложения (Prisma, Validation, Services, UI) для восстановления стабильности продакшена.
- [x] **[Build] Type Error `string[]` not assignable**: Исправлено через `npm dedupe` и синхронизацию версий Zod.
- [x] **[Build] Syntax Error in JSX**: Исправлены ошибки вложенности тегов в `page.tsx` и `OrderWizardLight.tsx`.

## Дополнения под SEO
- Внедрена вложенная структура роутинга: `apps/web/src/app/(main)/orders/[slug]/[orderSlug]/page.tsx`.
- Где первый сегмент `[slug]` — это слаг категории (например, `/cleaning/`), а второй `[orderSlug]` — слаг заказа.
- Все ссылки в приложении (лента, личный кабинет, отклики) обновлены для генерации новых иерархических путей.

### 2. Конфликт имен динамических сегментов Next.js (Routing Conflict)
**Описание:** Ошибка: `Error: You cannot use different slug names for the same dynamic path`.
**Причина:** В Next.js нельзя иметь на одном уровне папки `[slug]` и `[categorySlug]`. Двигатель роутинга требует единообразия имен переменных для динамических сегментов одного уровня.
**Решение:**
- Унифицированы имена папок. Теперь на первом уровне вложенности `orders/` используется только `[slug]`.
- Каноническая страница заказа перемещена в `orders/[slug]/[orderSlug]`, где `slug` (Level 1) интерпретируется кодом как слаг категории.
- Редиректор для старых ссылок оставлен в `orders/[slug]/page.tsx` с тем же именем переменной, что решило конфликт.

### 3. SEO Оптимизация и Структурированные данные (JSON-LD)
**Описание:** Отсутствие четкой навигационной цепочки в поиске Google.
**Решение:**
- Внедрены JSON-LD Breadcrumbs (хлебные крошки) на странице заказа.
- Реализованы серверные 301-редиректы. Если пользователь заходит по старому адресу `/orders/[id]` или по адресу с неправильной категорией, он автоматически перенаправляется на правильный иерархический URL. Это сохраняет вес страниц в поиске и предотвращает дублирование контента.

### 4. Исправление визуальных багов в карточке заказа
**Описание:** Номер заказа `№123` на детальной странице выглядел инородно перед заголовком и ломал верстку на мобильных устройствах при длинных названиях.
**Решение:**
- Номер заказа вынесен в правый верхний угол карточки в виде компактного бейджа (Purple Accent).
- Использован `ring-inset` и полупрозрачный фиолетовый фон (`bg-primary/10`) для премиального вида.

 ## [2026-04-22] Стабилизация CI/CD и внедрение планирования заказов
 
 ### 1. Исправление падения сборки (CI/CD Type Safety)
 **Описание:** Сборка проекта в GitHub Actions падала с ошибкой TypeScript (Exit Code 1).
 **Причина:** В интерфейс `OrderCardData` было добавлено обязательное поле `category.slug` для SEO-ссылок, но SQL-запрос в слое получения ближайших заказов (`order-geo.ts`) не выбирал это поле из базы данных.
 **Решение:**
 - Обновлен SQL-запрос в `order-geo.ts`: добавлен выбор `c.slug` через `JOIN`.
 - Результаты запроса типизированы и корректно маппятся в объекты `NearbyOrderCard`.
 - Проверено локальной сборкой `turbo build` — ошибки типов устранены.
 
 ### 2. Планирование времени исполнения (Order Scheduling)
 **Описание:** Заказчики не могли указать, когда именно нужно приступить к работе, что критично для услуг по расписанию.
 **Решение:**
 - В модель `Order` (Prisma) добавлено поле `executionDate`.
 - **UI (Создание):** В многошаговую форму `OrderWizardLight` на шаг «Детали» добавлен выбор даты и времени (`datetime-local`).
 - **UI (Лента):** На карточку `OrderFeedCard` добавлен яркий бейдж с иконкой часов и временем начала работ (если указано). Используется янтарный акцент (`amber-600`) для привлечения внимания.
 - **UI (Детали):** На странице заказа добавлена информация о сроке исполнения в основной блок параметров.
 - **Service Layer:** Обновлен `order.service.ts` для сохранения и корректного отображения даты во всех списках.
- **Validation:** Схема `createOrderSchema` дополнена полем `executionDate` с автоматическим приведением строки к объекту `Date` через `z.preprocess`.

## [2026-04-23] Внедрение универсальной геолокации и интеграция с DaData

### 1. Централизация логики геолокации (Architecture Refactor)
**Описание:** Логика определения местоположения была разбросана по компонентам, использовала разные стратегии и дублировала запросы к внешним API.
**Решение:**
- Создан универсальный хук `useLocation` в `shared/lib/hooks`. Он реализует каскадную стратегию: высокоточный GPS (3с) -> Браузерный Wi-Fi (4с).
- Удалено прямое использование сторонних IP-API на клиенте для повышения безопасности и надежности.

### 2. Единый сервис DaData (Shared Infrastructure)
**Описание:** Проект использовал DaData только для подсказок адресов, при этом IP-локация выполнялась через внешние API на клиенте.
**Решение:**
- Создан `src/shared/lib/dadata.ts` как единая точка взаимодействия с DaData на стороне сервера.
- Реализованы методы:
    - `detectCityByIp`: определяет город по IP, извлеченному из заголовков запроса (`headers()`).
    - `geolocate`: преобразует координаты GPS в название города через API DaData.
    - `suggestAddress`: унифицированные подсказки для всех форм проекта.
- Все существующие компоненты (например, `DadataAddressInput`) переведены на использование этого сервиса.

### 3. Интеграция выбора города в поиск (UX/UI)
**Описание:** Отсутствовала возможность фильтрации услуг по городу на главной странице.
**Решение:**
- Разработан компонент `CitySelector` с поддержкой:
    - Автоопределения города при первом входе.
    - Ручного поиска и выбора из списка активных городов в БД.
    - Персистентности выбора (через `localStorage` и URL-параметр `cityId`).
- Компонент интегрирован в `HeroSection`. Теперь кнопка «Найти» на главной странице автоматически учитывает локацию пользователя.

### 4. Поиск ближайшего города через PostGIS (Backend)
**Описание:** Необходимо было эффективно связывать "сырые" координаты пользователя с городами, присутствующими в системе.
**Решение:**
- В серверных экшенах реализован поиск через **PostGIS** оператор `<->` (KNN-search).
- Это позволяет мгновенно находить ближайший к пользователю город из базы данных на основе географических координат.
- Исправлены ошибки импортов и типизации, возникавшие при интеграции хуков в существующие виджеты.
