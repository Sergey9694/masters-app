## [2026-04-23] Стабилизация базы данных на VPS и оптимизация сидинга

### 1. Исправление сидинга и проблемы кодировки (Root Cause: PowerShell Encoding)
**Проблема:** При попытке сидинга на VPS база выбрасывала `Unique constraint failed` на поле `name` (город «Шахты»). Ошибка в логах показывала: `Key (name)=(??????????) already exists`.
**Расследование:** 
- Установлено, что при передаче данных через пайпы (например, `type seed.sql | ssh ...`) из Windows PowerShell, кириллица конвертировалась в знаки вопроса (`?????`) из-за несовпадения кодировок (UTF-8 vs CP1251).
- В результате города с одинаковым количеством букв (например, «Аксай» и «Шахты») превращались в идентичные строки `?????`, вызывая конфликт уникальности в базе.
**Решение:**
- **Base64 Transfer**: SQL-скрипты были закодированы в Base64 в памяти и раскодированы уже внутри Linux-контейнера на VPS, что полностью исключило влияние кодировки терминала.
- **Восстановление Администратора**: После сброса базы вручную пересоздан `admin@test.com` (пароль `password123`) с корректным хешем для доступа в админку.
- **Верификация**: Логи контейнера подтверждают точность работы PostGIS: координаты `lat=47.26, lng=39.86` теперь безошибочно определяются как «Аксай».

### 2. Полный сброс базы данных (Database Hard Reset)
**Описание:** На VPS накопились некорректные и дублирующиеся данные о городах (Москва, Казань), что вызывало ошибки уникальности и неправильное определение местоположения.
**Решение:**
- Выполнен полный сброс базы данных masters_db на VPS (Drop/Create).
- Это позволило начать «с чистого листа» без конфликтов внешних ключей.

### 2. Унификация логики сидинга (Seed Consolidation)
**Описание:** Процесс заполнения базы был раздроблен на несколько файлов (seed.ts, seed-cities.ts, seed-admin.mjs), которые конфликтовали друг с другом и содержали устаревшие данные.
**Решение:**
- Все скрипты объединены в единый идемпотентный prisma/seed.ts (и его JS-копию seed.mjs для продакшена).
- **Города**: Теперь берутся строго из cities.config.ts.
- **Админ**: Создается автоматически в том же процессе.
- **Очистка**: Внедрена логика деактивации (isActive: false) для всех городов, не входящих в актуальный список проекта.

### 3. Оптимизация запуска контейнера (Startup Process)
**Описание:** Файл startup.js в Docker-контейнере запускал сидинг городов по старым путям, что приводило к перезаписи правильных данных старыми (Москва).
**Решение:**
- Упрощен startup.js: удалены лишние и дублирующие вызовы .mjs файлов.
- Теперь за всё отвечает один вызов node prisma/seed.mjs, что гарантирует целостность данных при каждом перезапуске контейнера.

### 4. Исправление PostGIS координат
**Описание:** В некоторых скриптах использовалось поле lon вместо lng, что приводило к ошибкам в запросах ближайших соседей.
**Решение:**
- Все поля координат унифицированы к стандарту lat/lng во всех слоях приложения (Prisma, Seed, UI).

## [2026-04-23] Enterprise Geolocation & City Discovery System

### Проблема
1. **Неточность определения**: Использование только IP-баз давало ошибки в определении города (особенно на VPS или при использовании VPN).
2. **Рассинхрон данных**: Города в базе не имели единого идентификатора (типа FIAS ID), что затрудняло интеграцию с внешними сервисами (DaData).
3. **Отсутствие PostGIS**: Невозможно было найти "ближайший город" для пользователя, если его точное местоположение не числилось в базе.
4. **Стейт-менеджмент**: Хранение города в `localStorage` вызывало гидратацию и мешало SSR-компонентам (TopProviders) фильтровать данные на сервере.

### Решение
1. **Persistence (Cookies)**: Перенесли `cityId` из `localStorage` в Cookies (`httpOnly: false` для доступа с фронта, но с полной поддержкой SSR). Это убрало мерцание контента и позволило фильтровать данные в Server Components.
2. **Гео-каскад**: Реализовали трехуровневую систему определения:
    - `GPS/Browser` -> `PostGIS (Nearest Neighbor)` (самый точный).
    - `DaData IP API` -> `FIAS ID Match` (надежный фолбек).
    - `Spatial Proximity` (поиск ближайшего центра региона, если город не найден).
3. **UX Polish (GeoToastListener)**:
    - Реализован механизм сохранения уведомлений в `sessionStorage` перед перезагрузкой.
    - Внедрен глобальный компонент `GeoToastListener` в `RootLayout`, который отображает уведомления сразу после обновления страницы. Это решило проблему «мигающих» или исчезающих тостов.
4. **Архитектура данных (SSOT)**:
    - Создан `prisma/cities.config.ts` как единый источник истины.
    - Внедрен `fiasId` в модель `City` для 100% точности синхронизации.
    - Активирован PostGIS (`ST_SetSRID`, `ST_MakePoint`) для пространственных запросов.
5. **Инструментарий**:
    - `scripts/geo-sync.ts`: Универсальный инструмент обновления гео-данных через API DaData.
    - Обновлен `prisma/seed.ts` для автоматической настройки проекта под целевой регион (Ростовская область).

## [2026-04-23] Внедрение универсальной геолокации и интеграция с DaData

### 1. Централизация логики геолокации (Architecture Refactor)
**Описание:** Логика определения местоположения была разбросана по компонентам и дублировала запросы к внешним API.
**Решение:**
- Создан универсальный хук `useLocation` в `shared/lib/hooks`. Он реализует каскадную стратегию: высокоточный GPS -> Браузерный Wi-Fi.
- Удалено прямое использование сторонних IP-API на клиенте.

### 2. Единый сервис DaData (Shared Infrastructure)
**Решение:**
- Создан `src/shared/lib/dadata.ts` как единая точка взаимодействия с DaData на стороне сервера.
- Реализованы методы: `detectCityByIp`, `geolocate` (GPS в город), `suggestAddress`.

### 3. Интеграция выбора города в поиск (UX/UI)
**Решение:**
- Разработан компонент `CitySelector` с поддержкой автоопределения, ручного поиска и персистентности через Cookies.
- Компонент интегрирован в `HeroSection`.

## [2026-04-22] Стабилизация CI/CD и внедрение планирования заказов

### 1. Исправление падения сборки (CI/CD Type Safety)
**Причина:** В интерфейс `OrderCardData` было добавлено обязательное поле `category.slug`, но SQL-запрос в `order-geo.ts` его не выбирал.
**Решение:** Обновлен SQL-запрос с использованием `JOIN c.slug`.

### 2. Планирование времени исполнения (Order Scheduling)
**Описание:** Заказчики не могли указать, когда именно нужно приступить к работе.
**Решение:**
- В модель `Order` добавлено поле `executionDate`.
- В `OrderWizardLight` добавлен выбор даты и времени (`datetime-local`).
- На карточки добавлен бейдж с иконкой часов (`amber-600`).

## [2026-04-22] Иерархическая SEO-маршрутизация и структура категорий

### 1. Переход на структуру /orders/[categorySlug]/[slug]
**Описание:** Одиночные слаги `/orders/[slug]` не давали поисковикам понимания вложенности.
**Решение:**
- Внедрена структура `orders/[slug]/[orderSlug]`, где первый сегмент — слаг категории.
- Реализованы серверные 301-редиректы со старых ID и неправильных категорий.
- Исправлена ошибка `P2022` (отсутствие колонки executionDate в БД на сервере).

### 2. Конфликт имен сегментов Next.js (Routing Conflict)
**Проблема:** Ошибка `You cannot use different slug names for the same dynamic path`.
**Решение:** Унифицированы имена папок. Теперь на первом уровне вложенности `orders/` используется только `[slug]`.

## [2026-04-22] Исправление регистрации мастера и ленты заказов

### 1. Дублирование окна выбора файлов
**Причина:** Тег `<label>` вокруг `div` с `onClick` вызывал двойной триггер инпута.
