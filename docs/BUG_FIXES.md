## [2026-04-27] Баги в реализации Фазы 7 (чат)

### 1. AES-GCM: IV 16 байт вместо 16 → 12 байт (нарушение NIST)
**Причина:** Первоначально `randomBytes(16)` — но NIST SP 800-38D рекомендует 12 байт для AES-GCM (96 бит). 16 байт вызывают дополнительное хэширование IV, что ухудшает производительность и является нестандартным.
**Решение:** `apps/web/src/shared/lib/crypto.ts` — заменено на `randomBytes(12)`. Валидация ключа: `if (buf.length !== 32) throw`.
**Файл:** `src/shared/lib/crypto.ts`

### 2. AES-GCM: отсутствовала валидация длины ключа
**Причина:** `ENCRYPTION_KEY` без проверки — при ключе неправильной длины Node.js бросал невнятную ошибку `Invalid key length`.
**Решение:** Явная проверка `if (buf.length !== 32) throw new Error("ENCRYPTION_KEY must be 64 hex characters")` в `getKey()`.
**Файл:** `src/shared/lib/crypto.ts`

### 3. Тест crypto.test.ts: мутация глобального env без try/finally
**Причина:** Тест изменял `process.env.ENCRYPTION_KEY = undefined` без восстановления — при падении теста следующие тесты получали некорректное окружение.
**Решение:** Обёрнуто в `try/finally` для восстановления оригинального значения. Затем `ENCRYPTION_KEY` перенесён в `vitest.config.ts` env-блок, что полностью устранило проблему.
**Файл:** `src/shared/lib/crypto.test.ts`, `vitest.config.ts`

### 4. chat.service.ts: deleteMessage игнорировал отсутствующее сообщение
**Причина:** Метод делал `update` без предварительной проверки — если сообщение не существовало, Prisma бросала непонятную ошибку `Record to update not found`.
**Решение:** Добавлена явная проверка `if (!message) throw new Error("Сообщение не найдено")` перед update.
**Файл:** `src/services/chat.service.ts`

### 5. chat.service.ts: cursor не найден — тихий сбой
**Причина:** При cursor-based пагинации — если cursor-сообщение не существовало, Prisma делала запрос без `cursor` (silent fallback), возвращая первую страницу вместо ошибки.
**Решение:** Явная проверка существования cursor-сообщения с `throw` если не найдено.
**Файл:** `src/services/chat.service.ts`

### 6. chat.service.ts: CSV-инъекция через переносы строк
**Причина:** Текст сообщений мог содержать `\n` и `"` — при CSV-экспорте это ломало структуру файла.
**Решение:** Добавлено экранирование: `.replace(/\r?\n/g, " ").replace(/"/g, '""')` для каждого поля.
**Файл:** `src/services/chat.service.ts`

### 7. Vitest подхватывал Playwright e2e тесты
**Причина:** `vitest.config.ts` не имел `include` паттерна — Vitest обходил все `*.spec.ts` включая `e2e/chat.spec.ts`. Playwright'овский `test.describe` несовместим с Vitest-раннером, suite падал.
**Решение:** Добавлено `include: ["src/**/*.test.ts", "src/**/*.spec.ts"]` в `vitest.config.ts`.
**Файл:** `apps/web/vitest.config.ts`

### 8. ConversationParticipant: отсутствовал индекс по userId
**Причина:** Запросы `WHERE userId = ?` делали seq scan по всей таблице `ConversationParticipant`.
**Решение:** Добавлен `@@index([userId])` в Prisma-схему. Зафиксировано отдельным коммитом.
**Файл:** `apps/web/prisma/schema.prisma`

## [2026-04-26] Баги в форме создания объявления (ListingForm + ensureCityAction)

### 1. `Cannot read properties of undefined (reading 'user')` на `/my-listings`
**Причина:** `listingService.getByUser` не включал `provider.user` в Prisma select, хотя `ListingCard` обращался к `listing.provider.user.displayName`.
**Решение:** В `apps/web/src/services/listing.service.ts` в метод `getByUser` добавлен полный блок `provider` с вложенным `user`.

### 2. `Unique constraint failed on fields: (name)` при выборе «г Грозный»
**Причина:** `ensureCityAction` искал город по `{ name + region }`. Если в БД запись существовала без поля `region`, `findFirst` не находил её, после чего `create` падал на `@unique name`.
**Решение:** В `apps/web/src/shared/lib/ensure-city-action.ts` упрощён WHERE: `OR: [{ fiasId }, { name }]` — поиск только по `fiasId` ИЛИ `name` (который и так `@unique`).

### 3. Неявная ошибка «Выберите населённый пункт» при выборе области
**Причина:** Когда пользователь выбирал «Астраханская область» из подсказок, `handleAddressSelect` получал `cityName = undefined` и делал тихий `return` — форма показывала generic-ошибку валидации без объяснения причины.
**Решение:** Добавлен явный `setGeoError("Уточните до конкретного города или населённого пункта — область не подходит")` в `apps/web/src/app/(main)/my-listings/new/ListingForm.tsx`.

## [2026-04-24] Региональная SEO-архитектура URL и иерархическая маршрутизация

### Проблема
1. **Плоская структура**: URL вида `/orders/[categorySlug]/[orderSlug]` не учитывали географию на уровне пути. Поисковики (SEO) видели заказы в «куче», что мешало продвижению по запросам типа «услуги сантехника в Ростове».
2. **Дублирование контента**: Без явной привязки к городу в URL, одни и те же списки могли индексироваться по разным параметрам, размывая ссылочный вес.
3. **Навигационный разрыв**: Переключение города через фильтр меняло только Query Params, что не создавало новых посадочных страниц для поисковых роботов.

### Решение (Enterprise SEO Architecture)
1. **Новый стандарт пути**: Внедрена иерархия `/orders/[citySlug]/[categorySlug]/[orderSlug]`.
    - `/orders/[citySlug]` — лендинг города (все услуги в регионе).
    - `/orders/[citySlug]/[categorySlug]` — пересечение города и категории (самый ценный трафик).
    - `/orders/[citySlug]/[categorySlug]/[orderSlug]` — страница конкретного заказа с региональным контекстом.
2. **Умные редиректы (Canonical Mapping)**:
    - Реализована автоматическая переадресация (301) со старых путей `/orders/[categorySlug]/[orderSlug]` на новые с подстановкой реального города из БД.
    - Добавлены проверки на соответствие слагов (если заказ перемещен или слаги не совпадают — редирект на актуальный путь).
3. **SEO Оптимизация**:
    - Внедрена генерация метаданных (Metadata API) для каждого уровня вложенности.
    - Реализованы JSON-LD хлебные крошки (BreadcrumbList) для формирования красивых сниппетов в выдаче.
4. **UX & Filters**:
    - Компонент `OrdersFilters` переведен на `router.push(path)` вместо манипуляции `searchParams`. Смена города или категории теперь мгновенно переводит пользователя на соответствующую SEO-страницу.

### Файлы и инструкции
- **Сервисы**: `city.service.ts` (метод `getBySlug`), `order.service.ts` (инъекция слагов в выборки).
- **Роутинг**: Папка `apps/web/src/app/(main)/orders/[citySlug]`.
- **Фильтры**: `apps/web/src/widgets/OrdersFilters/ui/OrdersFilters.tsx`.

## [2026-04-24] Динамическая география и масштабирование на ЮФО + Новые регионы

### Проблема
1. **Ограниченность списков**: Ранее база наполнялась только крупными городами через сидинг. Пользователи из малых поселков, хуторов и станиц не могли создать заказ, так как их населенных пунктов не было в БД.
2. **Ригидность**: Добавление нового региона требовало ручного обновления конфигураций и перезапуска сидинга.
3. **Безопасность API**: Ключи DaData вызывались напрямую с клиента, что могло привести к краже лимитов.

### Решение (Архитектура «Динамической географии»)
1. **API Proxy**: Создан `/api/suggest/address/route.ts`. Поиск адресов теперь идет через сервер, где жестко зашит фильтр `locations` по 12 регионам (8 ЮФО + ДНР, ЛНР, Херсонская, Запорожская). Это гарантирует, что пользователь найдет адрес только в зоне нашего покрытия.
2. **Lazy Registration (`ensureCityAction`)**:
    - Внедрен механизм автоматической регистрации населенных пунктов при первом выборе.
    - Если города нет в БД, он создается «на лету» с генерацией уникального слага.
    - **Универсальная привязка**: Каждому новому пункту автоматически назначаются все корневые категории услуг (`CityCategory`), что исключает пустые списки услуг в новых городах.
3. **Безопасность**: API-ключи DaData удалены из клиентского кода. Все вызовы идут через `authActionClient`.

### Файлы и инструкции
- Подробный гайд по изменению географии: `docs/GEO_ARCHITECTURE.md`.
- Логика проверки регионов: `apps/web/src/shared/lib/ensure-city-action.ts` (перенесено из `order-creation/api/` в `shared/lib/` для переиспользования).

## [2026-04-23] Стабилизация базы данных на VPS и оптимизация сидинга

### 1. Исправление сидинга и проблемы кодировки (Root Cause: PowerShell Encoding)
**Проблема:** При попытке сидинга на VPS база выбрасывала `Unique constraint failed` на поле `name` (город «Шахты»). Ошибка в логах показывала: `Key (name)=(??????????) already exists`.
**Расследование:**
- Установлено, что при передаче данных через пайпы (например, `type seed.sql | ssh ...`) из Windows PowerShell, кириллица конвертировалась в знаки вопроса (`?????`) из-за несовпадения кодировок (UTF-8 vs CP1251).
- В результате города с одинаковым количеством букв (например, «Аксай» и «Шахты») превращались в идентичные строки `?????`, вызывая конфликт уникальности в базе.
**Решение:**
- **Base64 Transfer**: SQL-скрипты были закодированы в Base64 в памяти и раскодированы уже внутри Linux-контейнера на VPS, что полностью исключило влияние кодировки терминала.
- **Восстановление Администратора**: После сброса базы вручную пересоздан `admin@test.com` с корректным хешем для доступа в админку.
- **Верификация**: Логи контейнера подтверждают точность работы PostGIS: координаты `lat=47.26, lng=39.86` теперь безошибочно определяются как «Аксай».

### 2. Полный сброс базы данных (Database Hard Reset)
**Описание:** На VPS накопились некорректные и дублирующиеся данные о городах (Москва, Казань), что вызывало ошибки уникальности и неправильное определение местоположения.
**Решение:** Выполнен полный сброс базы данных masters_db на VPS (Drop/Create).

### 3. Унификация логики сидинга (Seed Consolidation)
**Описание:** Процесс заполнения базы был раздроблен на несколько файлов (seed.ts, seed-cities.ts, seed-admin.mjs), которые конфликтовали друг с другом и содержали устаревшие данные.
**Решение:**
- Все скрипты объединены в единый идемпотентный prisma/seed.ts (и его JS-копию seed.mjs для продакшена).
- **Города**: Теперь берутся строго из cities.config.ts.
- **Очистка**: Внедрена логика деактивации (isActive: false) для всех городов, не входящих в актуальный список проекта.

### 4. Исправление PostGIS координат
**Описание:** В некоторых скриптах использовалось поле `lon` вместо `lng`, что приводило к ошибкам в запросах ближайших соседей.
**Решение:** Все поля координат унифицированы к стандарту `lat/lng` во всех слоях приложения (Prisma, Seed, UI).

## [2026-04-23] Enterprise Geolocation & City Discovery System

### Проблема
1. **Неточность определения**: Использование только IP-баз давало ошибки в определении города (особенно на VPS или при использовании VPN).
2. **Рассинхрон данных**: Города в базе не имели единого идентификатора (типа FIAS ID), что затрудняло интеграцию с внешними сервисами (DaData).
3. **Отсутствие PostGIS**: Невозможно было найти «ближайший город» для пользователя, если его точное местоположение не числилось в базе.
4. **Стейт-менеджмент**: Хранение города в `localStorage` вызывало гидратацию и мешало SSR-компонентам (TopProviders) фильтровать данные на сервере.

### Решение
1. **Persistence (Cookies)**: Перенесли `cityId` из `localStorage` в Cookies (`httpOnly: false` для доступа с фронта, но с полной поддержкой SSR).
2. **Гео-каскад**: Реализована трехуровневая система определения:
    - `GPS/Browser` -> `PostGIS (Nearest Neighbor)` (самый точный).
    - `DaData IP API` -> `FIAS ID Match` (надежный фолбек).
    - `Spatial Proximity` (поиск ближайшего центра региона, если город не найден).
3. **UX Polish (GeoToastListener)**: Реализован механизм сохранения уведомлений в `sessionStorage` перед перезагрузкой. Внедрен глобальный компонент `GeoToastListener` в `RootLayout`.
4. **Архитектура данных (SSOT)**:
    - Создан `prisma/cities.config.ts` как единый источник истины.
    - Внедрен `fiasId` в модель `City` для 100% точности синхронизации.
    - Активирован PostGIS (`ST_SetSRID`, `ST_MakePoint`) для пространственных запросов.

## [2026-04-25] ESM import() vs createRequire — bcryptjs не найден в seed.mjs

### Root Cause
**Проблема:** `seed.mjs` использовал `await import("bcryptjs")` (ESM dynamic import). Node.js ES Module resolution **не использует `NODE_PATH`** — только CJS `require` его поддерживает. В Dockerfile bcryptjs устанавливался глобально, путь прописывался в `NODE_PATH`. Для `require()` работало, для ESM `import()` — нет. В итоге в `passwordHash` записывался plaintext пароль, и вход в админку был заблокирован.

### Решение
- В `apps/web/prisma/seed.mjs`: заменено `await import("bcryptjs")` на `createRequire` из встроенного модуля `"module"`.
- Убран опасный fallback `ADMIN_PASSWORD_HASH = ADMIN_PASSWORD`.
