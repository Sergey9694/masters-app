## КРИТИЧЕСКИЕ проблемы (утечка данных, открытые роуты)

### КРИТ-1: Нет `noindex` на странице `/chat`
- **Файл:** `apps/web/src/app/(main)/chat/page.tsx`, строка 8
- **Проблема:** `export const metadata = { title: "Сообщения — УслугиРядом" }` — без `robots: "noindex, nofollow"`. Несмотря на то что proxy.ts блокирует неавторизованных, Googlebot при crawl-error (попытка зайти без сессии) получит редирект на `/`, а не 401. Это технически безопасно, но при ошибке конфигурации proxy страница может быть проиндексирована.
- **Что нужно:** добавить `robots: { index: false, follow: false }` в metadata.
- **Приоритет:** ВЫСОКИЙ (не критический только потому что proxy работает — но это defence-in-depth).

### КРИТ-2: `/chat/[id]/page.tsx` — полное отсутствие metadata + нет noindex
- **Файл:** `apps/web/src/app/(main)/chat/[id]/page.tsx`
- **Проблема:** Нет экспорта `metadata` совсем. Страница наследует `title` из root layout: `"УслугиРядом — услуги рядом с вами"` — это публичный description с описанием платформы. При любом сбое proxy (mis-config) страница чата с именами участников могла бы быть проиндексирована.
- **Дополнительно:** `otherParticipant.user.firstName` используется в UI компоненте, но не в title. Пока безопасно, но отсутствие явного noindex — принципиальная проблема.

### КРИТ-3: Нет `robots.ts` (или `robots.txt`)
- **Проблема:** Файл `apps/web/src/app/robots.ts` отсутствует. Нет `public/robots.txt`. Поисковики могут пробовать обойти `/chat/*` роуты. Next.js по умолчанию не генерирует `Disallow: /chat/` без явного robots-файла.
- **Что нужно:** создать `apps/web/src/app/robots.ts` с явным `Disallow: /chat/`, `/dashboard/`, `/admin/`, `/settings/`, `/my-orders/` и т.д.

## ВЫСОКИЙ приоритет

### ВЫСОК-1: Отсутствует `Strict-Transport-Security` (HSTS) в nginx
- **Файл:** `nginx/default.conf`
- **Проблема:** Заголовок `Strict-Transport-Security` отсутствует. Конфиг слушает только порт 80 (HTTP), без HTTPS-блока. По требованиям CLAUDE.md (Part VI) обязателен: `Strict-Transport-Security: max-age=63072000; includeSubDomains`.
- **Примечание:** Если HTTPS терминируется на уровне внешнего балансировщика (облако), то HSTS должен добавляться там. Если nginx — конечная точка, это критично.

### ВЫСОК-2: Отсутствует `Content-Security-Policy` в nginx
- **Файл:** `nginx/default.conf`
- **Проблема:** `X-Content-Type-Options`, `X-Frame-Options`, `X-XSS-Protection` есть. Но `Content-Security-Policy` отсутствует. По требованиям CLAUDE.md (Part VI) CSP обязателен.

### ВЫСОК-3: Редирект неавторизованных — на `/`, а не на `/auth/login`
- **Файл:** `apps/web/src/proxy.ts`, строка 47
- **Проблема:** Неавторизованные пользователи, пытающиеся открыть `/chat` или `/chat/[id]`, редиректятся на `/` (главную страницу), а не на `/auth/login`. Это запутывающее UX — пользователь не понимает почему он не может зайти в чат.
- **Уточнение:** Внутри `chat/page.tsx` и `chat/[id]/page.tsx` есть дополнительная проверка `if (!user) redirect("/auth/login")`, но до неё дело не доходит — proxy уже редиректнул на `/`.
- **Что нужно:** В `proxy.ts` добавить специфичный редирект для `/chat/*` на `/auth/login` вместо `/`.

## СРЕДНИЙ приоритет

### СРЕДН-1: `AdminChatsPage` — нет metadata с noindex
- **Файл:** `apps/web/src/app/admin/(protected)/chats/page.tsx`, строка 8
- **Проблема:** `export const metadata = { title: "Чаты — Админ" }` без `robots: { index: false }`. Защита через layout гарантирует что неавторизованный не увидит страницу, но при сканировании Googlebot получит редирект на `/admin/login`. Явный noindex предпочтителен.

### СРЕДН-2: `AdminChatDetailPage` — полное отсутствие metadata
- **Файл:** `apps/web/src/app/admin/(protected)/chats/[id]/page.tsx`
- **Проблема:** Нет экспорта metadata вообще. Аналогично СРЕДН-1.

### СРЕДН-3: `cache_valid` в `/_next/static/` location без `add_header Cache-Control` перезаписи
- **Файл:** `nginx/default.conf`, строки 32-35
- **Проблема:** `proxy_cache_valid 200 1y` работает только при наличии `proxy_cache_path` в `http {}` блоке и активной зоны кэша. Без этого директива игнорируется. `add_header Cache-Control "public, immutable"` корректен, но `proxy_cache_valid` без зоны — мёртвый конфиг.

### СРЕДН-4: `X-XSS-Protection "1; mode=block"` устарел
- **Файл:** `nginx/default.conf`, строка 15
- **Проблема:** Заголовок `X-XSS-Protection` устарел и удалён из всех современных браузеров (Chrome 78+, Firefox). Современная замена — CSP. Оставлять можно, но он не даёт защиты. Лучше убрать во избежание confusion.

### СРЕДН-5: Нет проверки участника при доступе к `/chat/[id]` на уровне proxy
- **Файл:** `apps/web/src/proxy.ts` + `apps/web/src/app/(main)/chat/[id]/page.tsx`
- **Проблема:** proxy.ts пропускает авторизованного пользователя к любому `/chat/:path*`. Проверка "является ли пользователь участником этой беседы" происходит только в Server Component и chatService. Это архитектурно правильно (тяжёлая логика в DAL), но важно убедиться что chatService.getMessages() корректно выбрасывает 403 при отсутствии участия — проверено в `[id]/messages/route.ts`: да, `"Нет доступа к диалогу"` → 403. OK.

## Вердикт: НЕЛЬЗЯ мёрджить

**Блокирующие issues:** КРИТ-1, КРИТ-2, КРИТ-3 — отсутствие robots.ts и noindex на приватных страницах нарушает базовый принцип defence-in-depth. ВЫСОК-1 и ВЫСОК-3 снижают UX и не соответствуют требованиям CLAUDE.md.

**Минимальный план перед мёрджем:**
1. Создать `apps/web/src/app/robots.ts` с Disallow для `/chat/`, `/admin/`, `/dashboard/` и других приватных роутов.
2. Добавить `robots: { index: false, follow: false }` в `chat/page.tsx` и `chat/[id]/page.tsx` metadata.
3. В `proxy.ts` изменить редирект с `/` на `/auth/login` для приватных web-роутов.
4. Добавить в `nginx/default.conf` заголовок `Strict-Transport-Security` (если nginx — точка HTTPS-терминации).
5. Добавить CSP в nginx.
