# QA Audit — Чат (Фаза 7) | 2026-04-27

## КРИТИЧЕСКИЕ БАГИ (сломает VPS/продакшн)

### BUG-001: startup.js запускает Next.js standalone server.js, а НЕ server.ts с Socket.io
**Файл:** `apps/web/scripts/startup.js:170` → `const serverPath = path.join(__dirname, "apps", "web", "server.js")`
**Файл:** `Dockerfile:80` → `CMD ["node", "startup.js"]`

`startup.js` запускает `apps/web/server.js` — это стандартный Next.js standalone сервер без Socket.io.
Кастомный `server.ts` (который инициализирует Socket.io + Redis Adapter + Redis Bridge) в продакшн-образе **не вызывается вообще**.

`server.ts` используется только в dev через `"start": "tsx server.ts"`, но в Dockerfile `CMD ["node", "startup.js"]` обходит его полностью.

**Результат:** Socket.io в продакшне не работает. Чат будет грузить страницу, но WebSocket не подключится, real-time сообщения не придут.

**Фикс:** Скомпилировать `server.ts` в `server.js` при сборке и вызывать его из `startup.js` вместо Next.js standalone `server.js`. Или переписать `startup.js` на TS и добавить его в билд-пайплайн.

---

### BUG-002: tsx в devDependencies, но prod-скрипт `"start"` его использует
**Файл:** `apps/web/package.json:84` — `"tsx": "^4.21.0"` в `devDependencies`
**Файл:** `apps/web/package.json:9` — `"start": "tsx server.ts"`

`tsx` не включён в `dependencies`, значит в prod-окружении (если `npm install --omit=dev`) он недоступен. Это усугубляет BUG-001: даже если startup.js вызовет `tsx server.ts`, команда упадёт с `tsx: not found`.

---

### BUG-003: CSP блокирует WebSocket-соединение в браузере
**Файл:** `apps/web/next.config.mjs:38`
```
"connect-src 'self' https://telegram.org https://*.telegram.org"
```
`connect-src` не содержит `ws:` и `wss:`. Socket.io при установке WebSocket-соединения использует `ws://` или `wss://` схему. Браузер заблокирует upgrade по CSP.

**Результат:** Socket.io сделает fallback на HTTP polling (если разрешено `'self'`), или вообще не подключится. Real-time функциональность деградирует без явной ошибки в UI.

**Фикс:** Добавить `ws: wss:` в `connect-src`.

## ВЫСОКИЙ ПРИОРИТЕТ

### BUG-004: Заблокированный пользователь (chatBlockedAt) может читать сообщения
**Файл:** `apps/web/src/services/chat.service.ts:152-153`

`chatBlockedAt` проверяется только в `sendMessage`. В `getMessages` и на странице `/chat/[id]` блокировка не проверяется — пользователь с `chatBlockedAt != null` может открыть любой диалог и читать историю. Это нарушение логики модерации.

---

### BUG-005: Дедупликация сообщений в ConversationList при новом собеседнике
**Файл:** `apps/web/src/features/chat/ui/ConversationList.tsx:42-46`

При получении `new:message` от нового собеседника (диалог которого ещё нет в `list`) вызывается `router.refresh()`. Но состояние `list` при этом не обновляется немедленно — между ивентом и рендером новой записи есть задержка Server Action round-trip. Новый диалог появится только после полного рефреша страницы.

---

### BUG-006: Race condition при оптимистичном обновлении и socket ивенте
**Файл:** `apps/web/src/features/chat/ui/ChatWindow.tsx:195-210`

Логика: добавляем оптимистичное сообщение → отправляем → заменяем на реальное. При этом `new:message` сокет-ивент может прийти **до** того как `sendMessageAction` вернул результат. В этом случае `filtered.some(m => m.id === realMessage.id)` будет `true`, оптимистичное сообщение удалится, и всё выглядит правильно. Но если ивент **не приходит** (обрыв), а `sendMessageAction` возвращает успех — реальное сообщение добавляется без дублирования. Этот кейс корректен. Однако если `sendMessageAction` возвращает ошибку (блокировка, rate-limit), оптимистичное сообщение удаляется — `toast.error` показывается, UX корректен. Проблема низкая, но edge case существует.

---

### BUG-007: Rate limit — in-memory store не работает в multi-process / multi-instance
**Файл:** `apps/web/src/shared/lib/rate-limit.ts:13-14`

Rate limiter использует in-memory `Map`. В Docker при горизонтальном масштабировании (2+ реплики) каждый процесс имеет свой стор — лимит 30 сообщений/мин применяется per-instance, а не per-user глобально. В `server.ts` уже есть Redis, но rate-limit его не использует. Комментарий в коде: «В проде заменить на Redis» — но это не сделано.

## СРЕДНИЙ ПРИОРИТЕТ

### BUG-008: TypeScript ошибка в socket-events.ts
**Файл:** `apps/web/src/shared/lib/socket-events.ts:13`
```
error TS2304: Cannot find name 'MessageDTO'.
```
`MessageDTO` используется в типе `ServerToClientEvents`, но не импортируется из `@uslugi/shared-types` в этом файле (только реэкспортируется на строке 1 через `export type { MessageDTO }`). В самом файле тип недоступен как значение для аннотации. Строка `export type { MessageDTO }` делает реэкспорт, но не объявляет локальный тип.

---

### BUG-009: Infinite scroll — нет обработки курсора исчерпан
**Файл:** `apps/web/src/features/chat/ui/ChatWindow.tsx:47,162`

`hasMore` определяется как `initialMessages.length === 30` и `older.length === 30`. Если сервер вернул ровно 30 сообщений (последняя страница = ровно 30), `hasMore` будет `true`, и при скролле вверх будет сделан лишний запрос, который вернёт 0 результатов. После этого `setHasMore(0 === 30)` → `false`. Один лишний запрос — некритично, но может мигать лоадер.

---

### BUG-010: Онлайн-индикатор — hardcoded заглушка
**Файл:** `apps/web/src/features/chat/ui/ConversationList.tsx:99-101`

```tsx
<div className="absolute bottom-0 right-0 size-3 rounded-full bg-emerald-500 border-2 border-background" />
```
Зелёная точка показывается у **всех** собеседников всегда, независимо от реального онлайн-статуса. Это вводит пользователя в заблуждение.

### BUG-011: startConversation — гонка при одновременном создании
**Файл:** `apps/web/src/services/chat.service.ts:27-52`

`findFirst` → `create` не атомарно. Если два пользователя одновременно нажмут "Начать чат", может создаться два диалога с одинаковыми участниками и контекстом. Нет `unique` constraint или `upsert`. В БД нет индекса/ограничения на `(orderId, participantA, participantB)`.

### BUG-012: API endpoint POST /api/v1/conversations/[id]/messages не имеет rate limit
**Файл:** `apps/web/src/app/api/v1/conversations/[id]/messages/route.ts:41-77`

Server Action `sendMessageAction` имеет rate limit, но REST API endpoint не имеет. Мобильный клиент, использующий REST API, может флудить без ограничений.

---

### BUG-013: ConversationList — socket.off("conversation:update") без коллбека
**Файл:** `apps/web/src/features/chat/ui/ConversationList.tsx:51`

```ts
socket.off("conversation:update");
```
Вызов без второго аргумента (коллбек) снимает **все** обработчики этого события, включая те, что могли быть зарегистрированы другими компонентами. Правильно: `socket.off("conversation:update", handler)`.

---

### BUG-014: Нет обработки ошибки decrypt в mapMessage (runtime crash)
**Файл:** `apps/web/src/services/chat.service.ts:20`

```ts
text: m.deletedAt ? "[сообщение удалено]" : decryptText(m.text),
```

Если `ENCRYPTION_KEY` изменится (ротация ключа) или в БД окажутся legacy-незашифрованные сообщения, `decryptText` бросит исключение и весь `getMessages` упадёт с 500. В admin-страницах (`/admin/chats/[id]/page.tsx:89-92`) есть try/catch вокруг `decryptText`, но в `chatService.mapMessage` — нет.

## Вердикт: НЕЛЬЗЯ мёрджить

**3 критических бага блокируют продакшн:**

1. **BUG-001** — Socket.io физически не запускается в Docker (startup.js → server.js, а не server.ts). Чат не работает вообще.
2. **BUG-002** — tsx в devDependencies, prod-start его использует.
3. **BUG-003** — CSP блокирует WebSocket (`ws:` / `wss:` не в `connect-src`).

**Минимальный путь к деплою:**
1. Исправить Dockerfile/startup.js: скомпилировать `server.ts` → `server.js` (`tsc` или esbuild) и вызывать его вместо Next.js `server.js`.
2. Перенести `tsx` в `dependencies` или устранить его из prod-цепочки.
3. Добавить `ws: wss:` в `connect-src` в `next.config.mjs`.
