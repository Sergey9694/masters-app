# Karpathy Guidelines (Principles of Simplicity)

Эти принципы направлены на минимизацию сложности и предотвращение типичных ошибок ИИ при написании кода.

## 1. Сначала думай (Think Before Coding)

**Не делай предположений. Не скрывай непонимание. Озвучивай компромиссы.**

Перед реализацией:
- Явно формулируй свои предположения. Если есть сомнения — спрашивай.
- Если есть несколько вариантов решения, представь их все, не выбирай молча.
- Если есть более простой подход — скажи об этом.
- Если что-то непонятно — остановись и спроси.

## 2. Простота превыше всего (Simplicity First)

**Минимум кода для решения задачи. Никаких спекуляций.**

- Никаких фич сверх того, что просили.
- Никаких абстракций для кода, который используется один раз.
- Никакой "гибкости" или "настраиваемости", о которой не просили.
- Если 200 строк кода можно заменить на 50 — перепиши.

Задавай вопрос: "Счел бы опытный инженер это решение излишне сложным?" Если да — упрощай.

## 3. Хирургические изменения (Surgical Changes)

**Трогай только то, что необходимо. Исправляй только свой "беспорядок".**

При редактировании существующего кода:
- Не "улучшай" соседний код, комментарии или форматирование, если это не связано с задачей.
- Не делай рефакторинг того, что не сломано.
- Соответствуй существующему стилю, даже если ты сделал бы иначе.
- Если заметил несвязанный "мертвый" код — просто упомяни о нем, не удаляй сам.

## 4. Выполнение по целям (Goal-Driven Execution)

**Определи критерии успеха. Цикл до полной верификации.**

Превращай задачи в проверяемые цели:
- "Добавить валидацию" → "Написать тесты для невалидных данных, затем заставить их проходить".
- "Исправить баг" → "Написать тест, воспроизводящий баг, затем исправить".
- "Оформить рефакторинг" → "Убедиться, что тесты проходят до и после изменений".

Для многошаговых задач всегда составляй краткий план:
1. [Шаг] → проверка: [как проверить]
2. [Шаг] → проверка: [как проверить]
