## 🔴 ТОП-5 НЕПРЕЛОЖНЫХ ПРАВИЛ (важнее всего остального)

> Эти правила — ЗАКОН. Они стоят выше любых других локальных инструкций.

### 1. 💰 ЭКОНОМЬ ТОКЕНЫ — не трать впустую, но без ущерба качеству
- Читай только нужные строки файла, не весь файл целиком
- Один grep вместо чтения 10 файлов
- Сжатые отчёты: факты, не вода
- НО: никогда не экономь на проверках (`tsc`, тестах) — это ложная экономия

### 2. 🗄️ ИЗМЕНЕНИЯ БД — ТОЛЬКО через `prisma migrate dev`
- Никакого прямого SQL, никаких `$executeRaw` без миграции
- Каждое изменение `schema.prisma` → миграция с понятным именем
- Перед любым изменением схемы — консультация с владельцем

### 3. 🏆 ВСЕГДА ДАВАЙ РЕКОМЕНДАЦИЮ — никогда не перекладывай выбор на владельца
- Предлагай 2-3 варианта → **обязательно указывай лучший и объясняй почему**
- Владелец — не разработчик, он не может оценить варианты без твоей экспертизы
- Формат: "Вариант 1 / 2 / 3... 🏆 Рекомендую вариант N, потому что..."

### 4. 🧪 ОБЯЗАТЕЛЬНАЯ ПРОВЕРКА И ТЕСТ — стандарт 2026
"Код написан, но не проверен — значит, задача НЕ выполнена".
- **Static**: `npx tsc --noEmit` после КАЖДОГО изменения.
- **Unit/Logic**: `npm run test` (Vitest) для всех сервисов, утилит и Server Actions.
- **E2E/UX**: `npm run test:e2e` (Playwright) для критических путей (Auth, Chat, Orders).
- **Manual**: Визуальная проверка в браузере (Desktop + Mobile 375px).

### 5. 📜 ЦЕЛОСТНОСТЬ ИНСТРУКЦИЙ (Instruction Integrity)
- **Всегда проверяй правила**: В начале сессии ОБЯЗАТЕЛЬНО читай `CLAUDE.md`, `GEMINI.md` и `agent-protocol/SKILL.md`.
- **Синхронизация**: При изменении любого правила (например, протокола тестирования) агент ОБЯЗАН обновить ВСЕ три файла одновременно, чтобы избежать расхождения ("rule drift").
- **Наследование**: Локальные скиллы (`.agent/skills/`) дополняют общие правила, но никогда не отменяют их.

Это означает:
1. **Объясняй каждое решение** простым языком (зачем, какие альтернативы, какие риски)
2. **Никогда не принимай решения молча** — всегда уведомляй и советуйся
3. **Предполагай, что любое изменение может сломать прод** — проверяй ДО коммита
4. **Не усложняй** — выбирай простейшее решение, которое работает

## 🛑 Абсолютные правила (нарушение = СТОП)

### 1. НЕ пиши код сразу
Перед написанием любого кода выполни **Pre-flight Checklist**:

```
□ Понял ли я задачу? (переформулируй задачу своими словами)
□ Есть ли готовая библиотека для этого? (Build vs Buy)
□ Какие файлы затронутся? (список)
□ Какие edge-cases могу предвидеть? (список)
□ Может ли это сломать существующий функционал? (оценка риска)
□ Нужно ли спросить владельца? (если да — СТОП и спроси)
```

### 2. Build vs Buy — ВСЕГДА проверяй сначала
Перед написанием любого нового функционала:
1. **Поищи библиотеку** через `@ref`, `@context7` или `search_web`
2. **Оцени** по критериям:
   - Популярность (stars, weekly downloads)
   - Поддержка (последний коммит < 6 месяцев)
   - Размер бандла (bundlephobia.com)
   - Совместимость с Next.js 16+ / React 19
3. **Предложи владельцу 2-3 варианта** с плюсами/минусами
4. **ОБЯЗАТЕЛЬНО укажи свою рекомендацию** — какой вариант лучше и ПОЧЕМУ
5. **Дождись одобрения** перед установкой

Пример:
```
❌ "Я написал свой date-picker с нуля (400 строк)"

❌ "Есть 3 варианта: 1) ... 2) ... 3) ... Какой выбираете?"
   (НЕ заставляй владельца выбирать без рекомендации!)

✅ "Для date-picker есть 3 варианта:
   1. Shadcn DatePicker (Radix) — 0KB доп. бандл, наш стандарт
   2. react-day-picker — 8KB, больше кастомизации
   3. Свой компонент — не рекомендую (долго, багоопасно)
   
   🏆 Рекомендую вариант 1, потому что:
   - Мы уже используем Shadcn — не добавляет размер бандла
   - Уже стилизован под наш дизайн
   - Поддерживается сообществом и обновляется
   Одобряете?"
```

### 3. Автономность — решай сам, не беспокой владельца по мелочам

Владелец хочет видеть только результат. Беспокой его минимально.

**СТОП — обязательно спроси перед тем как делать:**
- [ ] Новая зависимость (`npm install`) — любая
- [ ] Изменение схемы БД (`schema.prisma`)
- [ ] Удаление файла, компонента, маршрута
- [ ] Изменение `next.config`, `docker-compose`, `.env`
- [ ] Изменение auth-логики или proxy.ts
- [ ] Архитектурное решение влияющее на > 5 файлов

**Решай сам, упомяни в финальном отчёте:**
- Выбор между двумя равнозначными реализациями (если нет явных рисков)
- Рефакторинг ≤ 3 файлов в рамках задачи
- Исправление типов, console.log, неиспользуемых импортов
- Добавление комментариев, именование переменных
- Порядок полей, форматирование

**Молча делай, не упоминай:**
- Исправление опечаток
- Добавление/исправление типов TypeScript
- Удаление console.log
- Форматирование

### 4. Коммит — только с одобрения владельца и ТОЛЬКО на РУССКОМ языке
**ЗАПРЕЩЕНО** делать `git commit` без явного разрешения.

**Требования к коммитам:**
1. **Язык**: ТОЛЬКО РУССКИЙ.
2. **Детализация**: Описывай ПОЧЕМУ сделаны изменения, а не только ЧТО. Коммит должен быть максимально развернутым и информативным.
3. **Группировка**: Объединяй правки по смыслу (infra, fix, feat, chore). Не смешивай всё в одну кучу.
4. **Чистота**: Перед коммитом ОБЯЗАТЕЛЬНО проверяй `git status`. Запрещено добавлять технический мусор: временные файлы тестов (`test-results`, `last-run.json`), избыточные отчеты агентов, логи.
5. **Процедура**:
   - Покажи `git diff --stat`.
   - Предложи развернутое сообщение коммита на русском.
   - **Жди явного "да" от владельца.**

Формат запроса:
```
## 💾 Готов к коммиту

**Изменённые файлы:**
- `path/to/file.ts` — [что изменилось]
- `path/to/file2.tsx` — [что изменилось]

**Сообщение коммита:**
`feat: [описание]`

Коммитим?
```

Исключение — только если владелец явно сказал «коммить сам» в рамках текущей задачи.

### 5. Проверяй ПОСЛЕ каждого изменения
После каждого атомарного изменения:
```bash
# 1. TypeScript компиляция
npx tsc --noEmit --pretty 2>&1 | Select-String "error TS"

# 2. Если есть ESLint
npx eslint apps/web/src/ --quiet 2>&1 | head -20

# 3. Если dev-сервер запущен — проверить нет ли ошибок в консоли

# 4. Если меняли Prisma schema
npx prisma validate
```

### 6. Изменения БД — ТОЛЬКО через миграции
**ВСЕ значимые изменения схемы базы данных** должны проходить через Prisma Migrate.

**ЗАПРЕЩЕНО:**
- Менять данные в проде через `prisma.$executeRaw` без миграции
- Добавлять/удалять поля вручную через SQL
- Пропускать `npx prisma migrate dev` после изменения `schema.prisma`

**Алгоритм изменения БД:**
```bash
# 1. Внести изменения в schema.prisma
# 2. Создать миграцию (ОБЯЗАТЕЛЬНО с понятным именем)
npx prisma migrate dev --name add_phone_to_user

# 3. Проверить сгенерированный SQL
# 4. Проверить что Prisma Client перегенерирован
npx prisma generate

# 5. Проверить что приложение работает
npx tsc --noEmit
```

**Перед любым изменением schema.prisma** — ОБЯЗАТЕЛЬНА консультация с владельцем:
```markdown
## 🗄️ Изменение БД: [Описание]

**Что меняется:** [Новое поле / таблица / индекс]
**Зачем:** [Бизнес-причина простым языком]
**Миграция:** `add_phone_to_user`
**Обратимо:** Да/Нет
**Риск для данных:** [Что может потеряться]

Одобряете?
```

### 7. Экономия токенов — НЕ трать впустую

**Токены стоят денег.** Работай эффективно:

**DO (делай):**
- Используй `grep_search` вместо чтения целых файлов, когда ищешь конкретную строку
- Читай только нужный диапазон строк файла (`StartLine`/`EndLine`)
- Группируй несколько правок в один `multi_replace_file_content`
- Одна команда с `&&` вместо нескольких отдельных
- Пиши сжато в отчётах — факты, не воду

**DON'T (не делай):**
- ❌ Не читай весь файл на 500 строк если нужна 1 строка
- ❌ Не запускай `tsc` на весь проект если менял 1 файл (хотя для проверки — допустимо)
- ❌ Не повторяй один и тот же `search_web` запрос дважды
- ❌ Не пиши длинные объяснения того, что и так очевидно
- ❌ Не вызывай MCP `context7` если уже знаешь API библиотеки
- ❌ Не создавай промежуточные файлы если можно сделать одним действием

**Принцип:** Минимум действий → Максимум результата.
Но **НИКОГДА** не экономь на проверках (tsc, тесты) — это ложная экономия.

### Пример рабочего процесса исследования
```

5. ref.search_documentation("react phone input mask Next.js")

```

## 📐 Глобальные правила проекта

Каждый агент ОБЯЗАН знать и соблюдать:

### Архитектура
- **FSD:** `shared → features → widgets → app` (строгая иерархия)
- **proxy.ts** вместо middleware.ts (Next.js 16+)
- **Server Components** по умолчанию, `'use client'` только для интерактивности
- **Server Actions** для мутаций, `route.ts` только для вебхуков

### Стек (Build vs Buy defaults)
| Категория | Решение | Пакет |
|-----------|---------|-------|
| UI | Shadcn UI (Radix) | shadcn/ui |
| Styling | Tailwind CSS + cn() | clsx + tailwind-merge |
| Icons | Lucide React | lucide-react |
| Forms | React Hook Form + Zod | react-hook-form + zod |
| Animations (simple) | Tailwind | tailwindcss |
| Animations (complex) | Motion | motion |
| Toasts | Sonner | sonner |
| Validation | Zod | zod |
| Server State | Server Actions + Prisma | prisma |
| Client State | Zustand (только UI) | zustand |
| URL State | searchParams | next/navigation |

### Безопасность
- **Zero Trust:** Валидация Zod на сервере для ВСЕХ входных данных
- **No `any`:** Использовать `unknown` + Type Guards
- **No localStorage** для токенов (только httpOnly cookies)
- **`select`** в каждом Prisma-запросе (не тянуть лишние поля)
- **Rate-limit** на всех публичных маршрутах
- **process.env** — только на сервере, через lazy getters

### Код
- **Файл > 200 строк** → рефакторинг
- **Комментарии и Документация** — русский язык
- **Сообщения коммитов** — ТОЛЬКО русский язык
- **Код** — английский язык (переменные, функции)
- **Серверная пагинация** для списков > 50 элементов
- **Нет `useMemo`/`useCallback`** без профилирования (React Compiler)

### Docker
- Multi-stage builds (Alpine)
- Секреты — runtime env, НЕ build args
- `ENV key="value"` (новый синтаксис)

## 🧪 Подробный протокол тестирования (Mandatory)

Агент ОБЯЗАН провести проверку по следующим уровням:

### L1: Статический анализ (TypeScript)
**Когда:** После каждой правки.
**Команда:** `npx tsc --noEmit --pretty`
**Критерий:** 0 ошибок в затронутых файлах и их зависимостях.

### L2: Юнит-тесты (Vitest)
**Когда:** При изменении логики сервисов (`src/services`), Server Actions или сложных утилит.
**Команды:**
- `npm run test` — запуск всех тестов.
- `npx vitest run path/to/file.test.ts` — запуск конкретного теста (экономия токенов).
**Критерий:** Все тесты (включая старые) проходят (Pass).

### L3: E2E Тесты (Playwright)
**Когда:** При изменении UI-компонентов, роутинга или критических пользовательских сценариев.
**Команды:**
- `npm run test:e2e` — запуск всех E2E тестов.
- `npx playwright test e2e/chat.spec.ts` — запуск конкретного сценария.
**Критерий:** Страница рендерится без ошибок, основные кнопки кликабельны, данные сохраняются.

### L4: Ручная верификация (Browser)
**Когда:** Любые изменения в UI/UX.
1. **Desktop**: Проверить выравнивание, ховеры, анимации.
2. **Mobile (375px)**: Убедиться, что ничего не вылезает за границы и кнопками удобно пользоваться.
3. **Empty States**: Как выглядит страница, если данных нет?
4. **Loading States**: Не прыгает ли верстка при загрузке?

## 🏁 Definition of Done (Критерии готовности)

Задача считается выполненной ТОЛЬКО если:
1. ✅ Код соответствует FSD-архитектуре.
2. ✅ Нет `any`, все типы валидны (`tsc` прошел).
3. ✅ Новые функции покрыты базовыми тестами (Vitest).
4. ✅ Существующие тесты (регрессия) проходят.
5. ✅ UI проверен на Desktop и Mobile.
6. ✅ `PROJECT_STATE.md` обновлен.

## 🔄 Обязательно после каждой задачи

**После завершения ЛЮБОЙ значимой задачи** агент ОБЯЗАН обновить документацию:

### 1. Всегда — обновить `docs/PROJECT_STATE.md`
- Статус фазы изменился → обнови таблицу статусов
- Новый/удалённый/переименованный ключевой файл → обнови раздел «Ключевые файлы»
- Добавлена миграция → добавь строку в «БД — последние миграции»
- Закрыт TODO → удали его из списка
- Обнови дату `🕓 Последнее обновление`

### 2. При исправлении багов — обновить `docs/BUG_FIXES.md`
- **Перед работой над багом**: прочитай файл — возможно, похожее уже решалось
- **После фикса**: добавь новую запись в формате:
  ```
  ## [YYYY-MM-DD] Краткое название бага
  **Причина:** [root cause]
  **Решение:** [что именно изменено и в каком файле]
  ```
- Фиксируй только реальные баги (не задачи и не рефакторинг)

### 3. Для крупных задач — отчёт в `docs/agent-reports/`
Формат имени: `[агент]-[YYYY-MM-DD].md`
Пример: `security-2026-04-26.md`

### 4. Для изменений БД — обновить `DEVELOPMENT_PLAN.md`
Найди нужный подпункт фазы, смени `❌` на `✅`.

**Не обновляй DEVELOPMENT_PLAN.md ради мелочей** — только статусы задач (✅/❌).
Полный DEVELOPMENT_PLAN.md читай только когда нужен глубокий контекст по конкретной фазе.
