## Критические принципы 2026

### 1. Data Access Optimization (Prisma profiling)
Медленная база — это смерть UX.
- Всегда используй `select`, чтобы не тянуть лишние поля.
- Для сложных выборок требуй `EXPLAIN ANALYZE` (через владельца или скрипты).
- Следи за N+1 запросами: используй `include` только там, где это оправдано.

### 2. Bundle Budgeting & RSC
- **RSC (React Server Components)**: Максимально выноси логику на сервер, чтобы не слать JS на клиент.
- **Dynamic Imports**: Загружай тяжелые компоненты (карты, редакторы) только тогда, когда они нужны.
- **Tree Shaking**: Проверяй, не тянет ли библиотека лишний код.

### 3. Core Web Vitals Mastery
- **LCP (Largest Contentful Paint)**: Оптимизируй загрузку первого экрана и `expo-image` в мобилке.
- **CLS (Cumulative Layout Shift)**: У компонентов всегда должны быть зарезервированные размеры (скелетоны).
- **FID (First Input Delay)**: Минимум блокирующих вычислений в основном потоке.

## Чеклист "Performance Audit"

### 1. Серверная часть (API/Database)
- [ ] Запрос выполняется быстрее 100мс?
- [ ] Используется ли кэширование для данных, которые редко меняются?
- [ ] Нет ли лишних `JOIN` в Prisma?

### 2. Фронтенд (Web/Mobile)
- [ ] Размер JS-бандла страницы < 100KB (gzip)?
- [ ] Все ли изображения имеют `priority` или `loading="lazy"`?
- [ ] Нет ли лишних `useEffect`, вызывающих цепные ререндеры?
- [ ] Используется ли `FlashList` для длинных списков в мобилке?

### 3. Infrastructure
- [ ] Включен ли Gzip/Brotli сжатие?
- [ ] Используется ли CDN для статики?
- [ ] Настроен ли `stale-while-revalidate` для API?

## Правила
1. **Не оптимизируй преждевременно**: Сначала измерь, потом фикси.
2. **Сначала сервер, потом клиент**: Лучшая оптимизация — та, которая не дошла до браузера.
3. **Используй MCP**: Ищи лучшие практики оптимизации через `search_web`.
4. Отчёт в `docs/agent-reports/performance-[дата].md`.
