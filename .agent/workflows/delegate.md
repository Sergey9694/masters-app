# 🎭 Workflow: Делегирование задач (Orchestrator)

Канонический workflow для команды `/delegate` и запросов вида "как оркестратор".
Он активирует роль Оркестратора для декомпозиции, безопасного планирования,
исполнения через профильные скиллы и финальной приемки сложных задач.

## Базовые правила

- Перед любым runtime-кодом выполняй Security Gate из `agent-security`, если задача затрагивает auth, PII, API, Server Actions, uploads, cookies, CSP, secrets, Docker/CI/CD, dependencies, Prisma, rate limits, API throttling или Idempotency-Key.
- Реальных параллельных субагентов запускай только когда текущие системные правила и запрос владельца это разрешают. Если запуск недоступен или неуместен, выполняй роли последовательно через чтение соответствующих `SKILL.md`.
- План показывай владельцу до исполнения, кроме случаев, когда владелец уже явно одобрил конкретный план.
- Не удаляй этапы Code Quality, QA, Security и Notary из этого workflow: это обязательная часть приемки.

## Шаги выполнения

1. **Инициализация**:
   - Прочитать `docs/PROJECT_STATE.md`.
   - Прочитать `.agent/skills/agent-protocol/SKILL.md`.
   - Прочитать `.agent/skills/agent-orchestrator/SKILL.md`.
   - При необходимости свериться с последними отчетами в `docs/agent-reports/`.

2. **Анализ задачи**:
   - Оценить масштаб задачи и затронутые модули.
   - Определить, затрагиваются ли runtime-код, безопасность, БД, зависимости, деплой или пользовательские данные.
   - Проверить, нужны ли дополнительные скиллы: `agent-architect`, `agent-security`, `agent-qa`, `agent-performance`, `agent-mobile-expert`, `agent-monorepo-master`, `agent-devops-cicd`, `agent-seo`, `shadcn-ui-mastery`, `playwright-testing`, `postgresql-optimization`, `docker-expert`, `sentry-expert`.

3. **Security Gate и Docs First**:
   - Для runtime/security-sensitive задач прочитать `.agent/skills/agent-security/SKILL.md` до финального плана.
   - Зафиксировать предварительный security verdict: `PASS`, `PASS_WITH_NOTES`, `BLOCKED` или `OWNER_DECISION`.
   - Если меняются API библиотек или есть сомнения в синтаксисе/поведении, свериться с актуальной документацией через доступные инструменты исследования.
   - При CRITICAL/HIGH риске остановиться до исправления или явного решения владельца.

4. **Декомпозиция и план**:
   - Разбить задачу на подзадачи: Research, Architecture, Security, Execution, QA, Documentation.
   - Назначить каждой подзадаче профильный скилл/роль.
   - Указать затронутые файлы, проверки, риски и ожидаемый результат.
   - Показать план владельцу и дождаться одобрения, если оно еще не было дано.

5. **Выполнение**:
   - Выполнять подзадачи через выбранные скиллы.
   - Соблюдать FSD-иерархию `shared -> features -> widgets -> app`.
   - Не менять `schema.prisma`, зависимости, auth/security, `.env`, `docker-compose`, `next.config` и крупную архитектуру без явного согласования.
   - Не откатывать чужие изменения; при конфликте работать поверх них или остановиться и спросить владельца.

6. **Ревью и контроль качества** (Role: Code Quality + QA + Security):
   - Активировать `agent-code-quality` для проверки архитектуры, типов, отсутствия `any`, мертвого кода и лишней сложности.
   - Активировать `agent-qa` для проверки сценариев, edge cases, регрессий и E2E-потребностей.
   - Активировать `agent-security` для финального security verdict, если задача затрагивала runtime/security-sensitive области.
   - Выполнить релевантные проверки: `npx tsc --noEmit`, unit/logic tests, Playwright/E2E или документальный diff для docs-only задач.

7. **Отчетность и синхронизация** (Role: Notary):
   - Активировать `agent-notary` для значимых задач, аудитов и многоагентных работ.
   - Подготовить отчет в `docs/agent-reports/`, если задача крупная или была выполнена через несколько ролей.
   - Обновить `docs/PROJECT_STATE.md` после значимой задачи.
   - Обновить `docs/BUG_FIXES.md` после исправления бага.
   - Обновлять `DEVELOPMENT_PLAN.md` только когда меняется статус фаз/задач, не ради мелких правок.

8. **Финальная сдача** (Role: Orchestrator):
   - Дать владельцу короткое резюме проделанной работы.
   - Перечислить измененные файлы и результаты проверок.
   - Отдельно отметить, что не запускалось или требует ручной проверки.
